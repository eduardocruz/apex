The probe with implementation: model emits a tool call, our server executes the tool, model sees the result, model writes the human answer. Click once, watch all turns.

The conceptual model — read once, never wonder again:

LayerWhat it doesWho provides it
The model (qwen2.5-7b-instruct)Reads the conversation, emits text or a JSON request to call a tool. Never executes anything.0G Compute Network (a GPU running open-source weights)
The tools (e.g. get_weather)The actual code that fetches data, hits an API, etc.We do. They live in server.js as TOOL_IMPLEMENTATIONS. In Phase 3 they will live in MCP servers.
The dispatcherReceives the model's tool-call request, runs the matching tool, pushes the result back to the model.We do. The agent loop in server.js.
The OpenAI tool-calling protocolThe wire format both sides agree on (tools, tool_calls, finish_reason).OpenAI defined it. 0G Compute supports it. The model was trained against it.

The model never reaches the network. We do. That's why MCP integration in Phase 3 is straightforward — only the dispatcher's executor changes.
